# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Catmagochi is a Tamagotchi-style virtual cat web app â€” a personal gift project with a hard deadline. It runs as an installable PWA so it works from a phone browser today, and is intended to later run on a Raspberry Pi with an attached screen (via Chromium in kiosk mode pointed at this same app, not a native rewrite). Keep the UI touch-friendly and flexible to small screen sizes, and avoid platform-specific APIs that wouldn't run in a plain browser (no native mobile modules, no App Store-only features).

## Commands

- `npm run dev` â€” start the Vite dev server
- `npm run build` â€” type-check (`tsc -b`) then production-build with Vite (also generates the PWA service worker/manifest via `vite-plugin-pwa`)
- `npm run lint` â€” run `oxlint`
- `npm run preview` â€” serve the production build locally
- `npm test` â€” run the frontend test suite once (Vitest); `npm run test:watch` for watch mode, `npm run test:coverage` to also emit a coverage report (thresholds enforced at 90% lines/statements/branches/functions in `vite.config.ts`'s `test.coverage` block â€” the `test` build task shares the rest of that config, e.g. the `__APP_VERSION__` define, since it's the same file).
- Relay server (optional, for the messaging and shared-pet features â€” see `server/README.md`): `cd server && npm install && RELAY_TOKEN=<secret> npm start`. Its own test suite is separate â€” plain `node --test`, no Vitest â€” via `cd server && npm test`; it spawns the real `server.js` as a child process against a temp `DATA_DIR` and drives it over real HTTP/WebSocket, matching the file's own "no framework" style. Vitest's `test.exclude` explicitly skips `server/**` so the two suites don't collide.

## Architecture

The game itself is client-only React + TypeScript. Pet state lives in `localStorage` â€” there's no server or API layer for gameplay. The exceptions are the optional messaging and shared-pet features (Â§Â§ below), which talk to a small self-hosted relay in `server/`.

- `src/types.ts` â€” `PetStats` (fullness/happiness/energy/cleanliness, each 0â€“100), `Mood`, `Stage` (`kitten`/`young`/`adult`), and `PetSave` (the persisted shape: name, stats, sleeping flag, lastUpdate timestamp, `growth`, `adoptedAt`, and the `total*` action counters â€” see `StatsWindow` below).
- `src/hooks/usePet.ts` â€” all game logic lives here:
  - Loads/saves a single `PetSave` under the `catmagochi-save-v1` localStorage key. `loadSave()` merges defaults (`{ growth: 0, adoptedAt: Date.now(), totalFeeds: 0, totalPlays: 0, totalCleans: 0, totalPets: 0, ...parsed }`) so saves from before any of these fields existed don't break â€” same pattern each time a new field is added to `PetSave`. `adoptedAt` on an upgraded old save is necessarily a best-effort "first loaded after the upgrade" date, not the pet's real original adoption date, since that was never recorded before.
  - Stat decay/regen is time-based, not tick-based: every 5s (`TICK_MS`) it recomputes stats from elapsed real milliseconds since `lastUpdate` using per-minute rates (`AWAKE_DECAY` vs `SLEEP_RATE`). This is also applied once on load, so stats "catch up" for time the app was closed (capped at 12 simulated hours so long absences don't require special-casing).
  - `deriveMood(stats, sleeping)` maps stats to a `Mood` by priority (sleeping > hungry > tired > dirty > sad > happy/content) â€” this is the single source of truth for both the on-screen mood text and which ASCII frame set renders.
  - `growth` (on `PetSave`, not `PetStats`) is earned only through positive actions, never passively over time: `feed` +3, `clean` +2, `pet` +1, `playGame` +3 (see `NudgePicker` below for what `play` means now). `src/data/growth.ts`'s `deriveStage(growth)` turns that into a `Stage`.
  - Actions (`feed`, `playGame`, `clean`, `toggleSleep`) are disabled while asleep except waking up. `pet()` is cooldown-gated (`PET_COOLDOWN_MS`, tracked via a ref, not persisted) and returns a boolean indicating whether it actually applied, so the UI only shows a reaction when the happiness bump really happened.
- `src/data/growth.ts` â€” `GROWTH_THRESHOLDS`, `deriveStage`, `STAGE_LABEL` (badge text), `GROW_MESSAGE` (banner text shown only on `young`/`adult` transitions, not the initial `kitten` state).
- `src/data/asciiCat.ts` â€” the cat is **analytic geometry rendered as text with supersampling**, not painted rows: per cell, 3Ã—3 sub-points are tested against the shape set (`makeSampler` â€” head ellipse, curved-edge ear triangles via `insideEar`'s eased interpolation, capsule whiskers via `nearSegment` with the cell `ASPECT` folded into distances, an eased A-line body) and coverage maps to ` `/`.`/`:` so curves dither smoothly instead of stair-stepping; pupil samples render as denser `#` with a cleared sparkle highlight. Look is matched to a reference terminal-pet screenshot the user supplied: huge notched ears (~top 30%), wide head, big oval negative-space eyes with chunky pupils, **no nose/mouth on the neutral face**, slanted whiskers with a gap from the head, tall A-line body, flat bottom, no tail. Idle animation is the pupils wandering across the eye holes (`PUPIL_DX` frames); mood mouths only exist for happy/sad/hungry/sleeping. `buildFrame(mood, blinkFrame, stage, idleFrame)` memoizes per combination (`DIMS`: kitten 40Ã—36 â†’ adult 60Ã—52). To inspect art changes, print frames in a terminal via `node --experimental-strip-types -e` importing `buildFrame` â€” much faster than screenshots for judging the silhouette. **Gotcha:** when testing visually, remember stats of 80 make `deriveMood` return `happy`, not `content` â€” comparing a browser render against a terminal print of a different mood looks like a rendering bug.
- `src/components/AsciiCat.tsx` + `AsciiCat.css` â€” the cat renders in a `<pre class="cat-sprite">` whose CSS (small `font-size` clamp + `line-height: 0.8`) makes the `':'` fill merge into a solid shape instead of reading as text. Two independent timers layer: the chained-`setTimeout` blink loop (resets on mood change; blink = eyes become cleared slits) and a `setInterval` pupil-wander (`IDLE_INTERVAL_MS` / `IDLE_FRAME_COUNT`) â€” plus a continuous CSS `gentle-bob` keyframe for idle sway. A floating **plain-ASCII** effect (`EFFECT` map â€” `*`, `?`, `;;`, `~`, `-.-`; no emoji anywhere in the app) plays above it, rendered from a single `glyphPop` state (`{ text, top, left, key }`) so cue reactions and the idle mood glyph share one rendering path: `popGlyph(text)` picks a fresh random `top`/`left` and bumps `key`, and `key` is what forces `.cat-effect`'s CSS `float-up` animation (a one-shot `forwards` fade, not a loop) to restart from the DOM on every pop rather than just re-triggering in place. Two things call `popGlyph`: `showCue(glyph)` (used by a successful `onPet()` for `<3`, and by the external `actionCue` prop â€” `{ type: ActionCueType } | null`, set by `App.tsx` â€” for the `ACTION_EFFECT` map's `feed`/`clean`/`sleep`/`wake` glyphs, plus the same `.reacting` bounce class), and a self-scheduling ambient loop (`AMBIENT_POP_MIN_MS`/`_JITTER_MS`, chained `setTimeout` like the blink loop) that keeps re-popping the idle `EFFECT[mood]` glyph at new random spots for as long as no cue is active â€” this existed first as a one-time-per-mood-change position, then was generalized to fully match the caption's "appear, fade, repeat" feel once asked for it too. **Gotcha:** the ambient loop is gated by `cueActiveRef` (a ref), not `cueGlyph` (state) â€” React runs every `useEffect` once on mount regardless of dependencies, so if `actionCue` were already non-null on the very first render, both the cue-triggering effect and the ambient effect would fire within that same commit; the ambient one would read `cueGlyph` from its own stale pre-update closure and clobber the cue's pop. A ref sidesteps this since `cueActiveRef.current = true` (set synchronously inside `showCue`, before the ambient effect runs) is visible immediately, with no closure staleness. This exact bug was unreachable through real usage (`App.tsx`'s `actionCue` state always starts `null`) but was still worth fixing rather than leaving latent. `showCue`'s revert timer explicitly falls back to `setGlyphPop(null)` when the mood has no `EFFECT` entry, rather than leaving a stale cue glyph on screen forever. Both `.cat-effect` and `.floating-caption` (`App.tsx`) use a stacked-`text-shadow` outline (four 1px offsets in `var(--bg)`) plus the existing glow, so the text stays legible now that it can land over any part of the card. Add new moods by extending the `Mood` union in `types.ts`, adding a case to `deriveMood`, and adding an eye/mouth stamp branch in `stampFace`.
- `src/components/NudgePicker.tsx` + `.css` â€” replaces `AsciiCat` in place (same panel chrome, imports `AsciiCat.css` for the shared `.ascii-screen` class) while `[PLAY]` is active. Not a mini-game (see "Play (a nudge, not a mini-game)" below for why) â€” just `NUDGE_OPTIONS` (`src/data/nudges.ts`) rendered as real `<button>`s, since the screen holds several independent tap targets rather than being one big tap target like `MessageView`'s. Picking one calls `onSend(text)`; a `[ CANCEL ]` button calls `onCancel()` to back out without applying or sending anything.
- `src/data/flavorText.ts` + `src/hooks/useFlavorText.ts` â€” `MOOD_LABEL` is the steady-state caption per mood; `useFlavorText` periodically (chained `setTimeout`, ~6-10s jittered) swaps in a random line from `FLAVOR_TEXT[mood]` (falling back to `GENERIC_FLAVOR`) for a few seconds before reverting. Resets whenever `mood` changes.
- `src/components/StatBar.tsx` â€” renders an ASCII block-character bar (`â–ˆâ–ˆâ–ˆâ–ˆâ–‘â–‘â–‘â–‘`, 10 segments) plus a 4-letter `code` label, colored by low/mid/high threshold; accepts an `isPulsing` prop (now a `Set<keyof PetStats>` in `App.tsx`, since `playGame` can pulse multiple stats at once) for the brief glow shown right after an action changes that stat.
- `src/components/StartScreen.tsx` â€” a brief boot splash shown before anything else, every time the app loads (not just first run). Exists because the OS/browser's own auto-generated PWA splash (built from the manifest â€” icon centered on `background_color`) can't be turned into a real designed screen; it's icon+color and nothing else, controlled before any of our JS runs. This component is the practical stand-in: it renders the instant React takes over, so for an app this size it reads as the actual loading screen. Pure `setTimeout` choreography, no real loading happening â€” `START_DISPLAY_MS` (900ms) then adds a `fading` class (CSS opacity transition), and `START_TOTAL_MS` (1200ms total) later calls `onDone`, which is what tells `App.tsx` to reveal `NameScreen`/the game underneath. Uses the same "ref updated during the render body" pattern as `usePet.ts`'s `saveRef` (`onDoneRef`) so the timer effect can have an empty dependency array and not restart if `onDone`'s identity changes between renders.
- `src/App.tsx` â€” top-level flow: `showStart` gates on `StartScreen` first (see above), then shows `NameScreen` until a save exists (first run), otherwise renders the game screen driven by `usePet` and `useFlavorText`. `playPickerOpen` state swaps `AsciiCat` for `NudgePicker` in place when `[PLAY]` is pressed (all other action buttons disabled meanwhile). A `prevStage` ref + `useEffect` detects stage transitions and shows a transient `.grow-banner`.
  - Every action button (`FEED`/`CLEAN`/`SLEEP`/`WAKE`) and a successful pet also call `triggerCue(type)` (sets `AsciiCat`'s `actionCue` prop, always as a fresh `{ type }` object rather than a bare `ActionCueType` string â€” pressing the same action twice in a row needs the prop *reference* to change so `AsciiCat`'s `useEffect(() => {...}, [actionCue])` re-fires; a repeated identical string would fail React's `Object.is` check and silently skip re-triggering the cue) and `maybeShowActionFlavor(type)`. The latter is a flat `ACTION_FLAVOR_CHANCE` (25%) roll that, on a hit, temporarily overrides `flavorText` (from `useFlavorText`) with a bonus line from `ACTION_FLAVOR[type]` (`src/data/flavorText.ts`) â€” plain local `useState`/`setTimeout`, doesn't touch `useFlavorText`'s own internals. Two independent "extra delight" layers (glyph+bounce on the cat, bonus caption text) fire from the same action, but are otherwise unrelated mechanisms.
  - There's no persistent, always-visible mood caption anymore â€” deliberately so, per the "one static place is boring" feedback that motivated this. `captionText` (name + whichever of `actionFlavor`/`flavorText` is current) is watched by a `useEffect`; every time its *value* changes, a `captionPop` object is recreated with a fresh random `top`/`left` (bounded to roughly the upper half of the card, away from the stat bars/buttons) and an incrementing `key`. The `key` is what forces React to unmount/remount the `.floating-caption` div so its CSS `caption-float` animation (fade in, hold, drift up + fade out over 2.6s, `animation-fill-mode: forwards`) restarts from the DOM rather than just visually re-triggering on the same node. Between value changes the last pop just sits there faded to `opacity: 0` (harmless, cheaper than adding cleanup logic to unmount it) until the next change replaces it outright.
  - The `.stage-label` badge (`[KITTEN]` etc) is tappable (`role="button"`, `aria-expanded`) and toggles `showGrowthProgress`, which reveals a `StatBar` reused as a growth-progress bar (`code="GROW"`) sized via `src/data/growth.ts`'s `growthProgress(growth)` â€” progress is *within the current stage's band*, not raw `growth`/threshold, so a young cat just past the young threshold reads as ~0% toward adult rather than >100% against a threshold it already cleared. `adult` has no `nextStage`, so the bar is replaced with a static "fully grown!" line instead of a maxed-out bar (there's nothing further to progress toward).
  - The pet's name is tappable too, opening `StatsWindow`. **Gotcha:** it's `<h1><button className="name-button" onClick={...}>{save.name}</button></h1>`, not `role="button"` on the `<h1>` directly â€” the latter was tried first and silently overrides the element's accessible role from `heading` to `button`, breaking any `getByRole('heading', ...)` query for the pet's name (including existing tests). Nesting a real `<button>` inside preserves the `<h1>`'s heading semantics (its accessible name is still computed from the button's text) while getting native keyboard activation for free, unlike the `role="button"` + manual `onKeyDown` pattern used everywhere else in this codebase (`AsciiCat`, `MessageView`, the stage badge above) â€” those all wrap non-native elements (`div`/`p`) that have no built-in Enter/Space handling, so they need it; a real `<button>` doesn't.

### Visual theme

Dark purple retro-terminal look (deliberate single theme, not responsive to system light/dark preference â€” see `src/index.css`): monospace font globally, CSS variables for the palette (`--bg`, `--text`, `--border`, `--primary`, `--glow`, etc.), bordered sharp-cornered boxes instead of rounded cards, bracket-style buttons (`[FEED]`). Keep all UI text and effects to plain ASCII/CSS â€” no emoji â€” to stay consistent with this theme.

### Messaging ("send from home")

Lets the user push short messages to the app from anywhere, via a self-hosted relay (not a managed third-party service â€” see `server/README.md` for why and how to deploy). Same panel-swap pattern as `NudgePicker`.

- `server/server.js` â€” standalone Node process (own `package.json`, not part of the Vite build/frontend). Plain `http` + `ws`, no framework. `GET /ws?token=` upgrades to a WebSocket after checking `RELAY_TOKEN`; `POST /send` (also token-checked) queues a message and broadcasts it to connected sockets. Undelivered messages persist to `server/messages.json` (or `$DATA_DIR/messages.json` in production, meant to be a mounted volume) and replay to a client as soon as it connects, so messages sent while the device is offline still arrive. Clients ack (`{type:'ack', id}`) once a message is actually displayed, not just received. `/send` also accepts an optional `kind` field (checked against `MESSAGE_KINDS`, currently just `'nudge'`); an unrecognized `kind` is silently dropped rather than rejecting the send, since it's a decorative/rendering hint, not something worth failing a request over.
- `src/hooks/useMessages.ts` â€” no-ops entirely (never opens a socket) if `VITE_RELAY_URL`/`VITE_RELAY_TOKEN` aren't set at build time, so the app works standalone before/without the relay. Otherwise connects and auto-reconnects with a chained-`setTimeout` backoff (1s â†’ 30s cap), same style as the blink loop in `AsciiCat.tsx` and `useFlavorText.ts`. Holds a local FIFO `messages` queue. Also exposes `send(text, kind?)`, a fire-and-forget `POST /send` (matching `sender.html`'s own send behavior) â€” unlike care events, a nudge isn't part of a replayed stat log, so there's no outbox to retry from; a send attempted while offline just fails silently.
- `src/components/MessageView.tsx` â€” reuses `.ascii-screen` from `AsciiCat.css` (same "one little screen, different content" pattern as `NudgePicker`). Tapping calls `dismiss(id)` (acks + removes from the local queue); `App.tsx` also calls `usePet`'s `receiveMessage()` (+5 happiness, no growth â€” receiving a note isn't "care" the way feeding/playing/cleaning is) on dismiss, *unless* the message is `kind: 'nudge'`, since a nudge already rewarded the shared cat via its `play` care event at send time â€” see "Play (a nudge, not a mini-game)" below.
- In `App.tsx`, render priority in the shared panel slot is `NudgePicker` > `MessageView` > `AsciiCat` â€” an incoming message never interrupts an open nudge picker, it just waits. `actionsDisabled` includes `messages.length > 0`.
- `sender.html` (repo root) â€” a standalone plain HTML/JS page (no build step, not part of the React app or PWA) for actually sending *freely-typed* messages: a textarea posting to `RELAY_URL/send`. Deliberately kept out of the main app bundle so there's no path for the message recipient to stumble into a general-purpose "send" UI â€” `NudgePicker` is a separate, much narrower affordance (a few canned lines, no typing) added deliberately to the main app; it doesn't reopen that door.
- The `RELAY_TOKEN`/`VITE_RELAY_TOKEN` shared secret is not real auth â€” it ends up in the client bundle, so treat it as "keeps random strangers out," not a security boundary. See `server/README.md` for rotating it.

### Shared pet (care event sync)

Lets two devices treat one `PetSave` as the same cat instead of each having an independent local one. Care actions (feed/clean/pet/play) sync between devices over the same relay messaging uses, replayed as an event log rather than raw stat numbers, so simultaneous offline actions from both sides merge instead of one clobbering the other. `sleeping` is deliberately **not** synced â€” it's a local UI toggle, not a care action, and syncing it would mean one device's nap disabling the other's buttons.

- `usePet.ts`'s `applyCareEvent(save, type)` is the single pure function every care delta (feed/clean/pet/play) goes through â€” extracted from what were previously four separate inline `setSave` updaters, so a local action and a replayed remote event produce identical results. `usePet(onCareEvent?)` takes an optional `(id, type) => void` callback invoked whenever a local action actually applies (never on a no-op, e.g. while sleeping or on `pet()`'s cooldown) â€” `id` is `crypto.randomUUID()`, generated client-side rather than server-side (unlike messages), specifically so it can be added to `appliedEventIds` (a `Set<string>` ref) *before* the event is ever sent â€” the relay broadcasts to every connected client including the sender, so without this a device would eventually receive its own action echoed back and double-apply it. `applyRemoteEvent(id, type)` is the inbound counterpart: checks `appliedEventIds` for dedup (covers both echoes of this device's own events and events redelivered after a reconnect that were already applied) and, unlike local actions, is **not** gated on `sleeping`. `appliedEventIds` is in-memory only, not persisted â€” the accepted gap is an action taken right before the app closes, before its ack reaches the relay, being replayed and double-applied once on the next reconnect; judged not worth persisting/pruning a growing id set for a personal pet, not a ledger.
- `src/hooks/useCareEvents.ts` mirrors `useMessages.ts`'s reconnect/backoff WebSocket shape (its own separate connection, not shared with messaging â€” merging them would still need per-frame-type filtering on both sides anyway), but differs in two ways. Incoming events are silent and applied immediately via the `onEvent` callback then acked, rather than queued for a person to dismiss like messages are. Outgoing events go through a small `localStorage`-backed outbox (`catmagochi-care-outbox-v1`) instead of a bare fire-and-forget POST â€” the whole point of an event log is that offline actions aren't lost, so `emit(id, type)` queues to the outbox first and only removes an entry once its POST actually succeeds; a failed/impossible send (offline) just leaves it queued. Flushing is opportunistic: attempted immediately in `emit()`, and again whenever the WebSocket reconnects, reusing that connection's own backoff pacing instead of adding a second retry timer.
- `App.tsx` wires the two hooks together, which has a chicken-and-egg problem: `usePet`'s `onCareEvent` needs `useCareEvents().emit`, and `useCareEvents`'s `onEvent` needs `usePet`'s `applyRemoteEvent` â€” each hook's callback is only known once the *other* hook has already been called. Solved with a `handleRemoteCareEventRef` ref, assigned fresh every render right after `pulse`/`triggerCue`/`maybeShowActionFlavor` are defined (mirroring the same "ref updated during the render body, not inside an effect" pattern `usePet.ts` uses for `saveRef`), plus a closure for the other direction (`usePet(() => careEvents.emit(...))` referencing `careEvents` before its own `const` line â€” safe because the closure isn't *invoked* until a later action, well after `careEvents` is assigned). A successfully-applied remote event gets the same reactions a local action does: `CARE_EVENT_STATS` maps each `CareEventType` to the stat bars it pulses (mirroring `applyCareEvent`'s deltas), and `feed`/`clean`/`play` also get the matching `AsciiCat` glyph cue via `triggerCue`. `pet` skips the glyph cue (no compelling reaction was designed for it) but still pulses happiness, and gets a bonus caption chance via `maybeShowActionFlavor` â€” `play` deliberately doesn't, matching how a *local* `playGame`/nudge send never calls `maybeShowActionFlavor` either, since the nudge's own text is already the "bonus" moment.

### Play (a nudge, not a mini-game)

`[PLAY]` used to launch a `YarnGame` reflex mini-game (3 timed rounds of "tap the target before it disappears"), with `playGame(hits)` scaling the reward by how many rounds you caught. That's gone. The idea came from wanting PLAY to feel like *actually taking care of a real cat together* rather than administering a stat via a button â€” see the design conversation that led here: photos were considered and rejected (too sensitive to send through a relay whose token "is not real auth"), a same-screen or real-time two-person interaction was considered and rejected (the two people using this app are apart during the day, not sitting at one screen), and a plain flat button was rejected as still feeling like "mindlessly pressing buttons." What landed: tapping `[PLAY]` opens `NudgePicker` (a few canned, sentence-case lines â€” no typing) instead of a mini-game; picking one both applies a flat `play` care-event reward to the shared cat (same mechanism as feed/clean/pet, no more hit-count scaling) *and* sends the picked line to the other device as a `kind: 'nudge'` message via `useMessages`'s `send()`. `App.tsx`'s `handleSendNudge` does both in one call, then closes the picker.

- `applyCareEvent`'s `'play'` case: `happiness +10`, `growth +3`, with light `energy -5`/`fullness -3`/`cleanliness -3` costs so it isn't a strictly-better substitute for feed/clean. Weighted toward happiness since that's the point of it now.
- On the receiving device, the nudge shows up two ways at once: the `play` care event pulses stats and pops the `*purr*` glyph on the cat (ambient, automatic, same path any synced care event takes), while the actual message ("Thinking of you") arrives separately through the existing messaging channel and shows in `MessageView` like any other note. Dismissing it does *not* also trigger `receiveMessage()`'s generic +5 happiness bonus â€” that bonus exists for freely-typed notes, which have no care event of their own; a nudge already got its reward at send time, and double-counting it would be a real (if minor) stat-sync inconsistency.
- If the relay isn't configured (solo use, no partner device), `[PLAY]` still works â€” the care event applies locally same as any other action, and `send()` (from `useMessages`) is simply a no-op. There's no separate "PLAY is unavailable offline" state; it degrades the same way messaging and shared-pet sync already do elsewhere in this app.
- `src/data/nudges.ts`'s `NUDGE_OPTIONS` are deliberately sentence-case, not the app's usual ALL-CAPS bracket-button style â€” they're meant to read as a genuine note, not a system command.
- Server-side (`server/server.js`): `POST /care-event` and the WebSocket broadcast/replay/ack extension follow the same shape messaging already established (`EVENTS_FILE`/`pendingEvents`/`persistEvents()` mirror `messages.json`/`pending`/`persist()`; `broadcastEvent()` mirrors `broadcast()`) â€” see `server/README.md`'s "Shared pet" section for the wire format.
- This is a pragmatic delta-replay approach, not a full CRDT: if two devices independently push the *same* stat toward its clamp boundary while both offline (e.g. both feed a nearly-full cat), replaying in a different order on each device could in principle leave them a point or two apart at the boundary. `growth` and the `total*` counters are unaffected since they're purely additive with no clamp. Accepted as out of scope given the stakes (a personal gift project, not a shared ledger) â€” flagged here rather than silently ignored.

### Menu

`src/components/Menu.tsx` is a full-card overlay (`position: absolute; inset: 0` inside `.game`, which needs `position: relative`), opened via the `[MENU]` button in the top-right corner (`App.tsx`, disabled while `gameActive` â€” checking the menu mid-minigame isn't blocked from a message being shown, only from an active game). Internal `view: 'root' | 'history' | 'update' | 'settings'` state, reset to `'root'` whenever it opens. The root view is a data-driven `MENU_ITEMS` array specifically so new entries are a one-line addition, not a restructure â€” `history`/`update`/`settings` were each added this way.

`src/hooks/useMessageHistory.ts` persists dismissed messages to `catmagochi-message-history-v1` (capped at 50, newest first) â€” separate from `useMessages`' in-flight queue, which only holds undismissed messages. `App.tsx`'s `handleDismissMessage` calls `record(message)` alongside the existing `dismiss`/`receiveMessage`/pulse calls, so history capture piggybacks on the same dismissal path rather than being a separate flow.

### Stats window

`src/components/StatsWindow.tsx` is a *separate* full-card overlay from `Menu` (own `statsOpen` state in `App.tsx`, opened by tapping the pet's name â€” see the `<h1>` gotcha above), not another `Menu` view â€” it needs `save`/`mood`/`stage` as direct props rather than living inside Menu's own state machine, and there was no obvious shared concern to justify coupling it to Menu's `view` union. Reuses `Menu.css`'s `.menu-overlay`/`.menu-panel`/`.menu-title`/`.menu-close` for the shared chrome (same "shared CSS file, component-specific rows" pattern as `AsciiCat.css`'s `.ascii-screen` reused by `NudgePicker`/`MessageView`), plus its own `StatsWindow.css` for the `.stats-row` label/value layout. Renders a flat `[label, value][]` array of rows (stage, mood, growth, `daysAgo(adoptedAt)`, the four stats rounded to whole percentages, then the four `total*` counters from `PetSave`) rather than hand-writing each row's JSX, so adding a new stat later is a one-line array entry.

### Push notifications

Three notification types, only two of which are real push â€” see the "settings view" `menu-settings-note` text in `Menu.tsx` for the user-facing version of this distinction:

- **`message`** â€” a message arrived via the relay. Real server push (extends the messaging relay above).
- **`update`** â€” a new GitHub release was published. Real server push, but **manually triggered**, not polled â€” see `server/README.md`'s `/push/notify-update`. The server deliberately doesn't poll GitHub's releases API on a timer (rate-limit/complexity not worth it for a one-person repo); run the curl command yourself right after `gh release create`.
- **`attention`** â€” a stat (fullness/energy/cleanliness/happiness) dropped below 25. **Local only** â€” pet state lives in the browser's `localStorage`, the server has no way to know about it, so this can never be true push. Only fires while the tab/app is alive (foreground or recently backgrounded by the OS), never from a fully-closed state. This is a real, permanent capability gap versus the other two types, not a "not implemented yet."

Client-side pieces:

- `src/hooks/useNotificationSettings.ts` â€” `{ global, message, attention, update }` booleans persisted to `catmagochi-notification-settings-v1`. Defaults to `global: false` (opt-in only â€” never presume the user wants an OS permission prompt) with the three per-type flags defaulting `true` so turning `global` on doesn't also require re-enabling every type.
- `src/hooks/usePushSubscription.ts` â€” owns the actual `PushManager` subscribe/unsubscribe lifecycle, reacting to `settings.global`/`message`/`update` changes (not `attention`, since that's never sent to the server). No-ops (`status: 'idle'`) if `VITE_RELAY_URL`/`VITE_RELAY_TOKEN`/`VITE_VAPID_PUBLIC_KEY` aren't all configured, matching `useMessages`'s "standalone without the feature" pattern. Derives the HTTP(S) subscribe/unsubscribe endpoint from the same `VITE_RELAY_URL` used for the WebSocket (`wss://`/`ws://` â†’ `https://`/`http://` via `.replace(/^ws/, 'http')`) rather than needing a second env var. A `syncing` ref guards against overlapping in-flight syncs if `settings` changes again before the previous fetch/subscribe finishes â€” the newer change is simply dropped rather than queued, since it'll naturally be picked up on the next actual settings change or remount.
- `src/hooks/useAttentionNotifications.ts` â€” the local-only `attention` type. Watches `stats`/`sleeping`/`settings`, reuses `usePet`'s own critical-stat priority order (fullness > energy > cleanliness > happiness, matching `deriveMood`), and rate-limits itself to one notification per 30 minutes (`MIN_NOTIFY_INTERVAL_MS`, tracked via a ref) so it doesn't spam on every 5s tick while a stat stays low.
- `src/components/Menu.tsx`'s `settings` view â€” a global toggle plus three per-type toggles (each `disabled` while global is off, but still reflecting their own stored value underneath). Shows `PUSH_STATUS_TEXT[pushStatus]` for `unsupported`/`denied`/`error` states (nothing shown for `idle`/`subscribed`/`unsubscribed` â€” those aren't actionable by the user).
- `App.tsx` owns `useNotificationSettings`/`usePushSubscription`/`useAttentionNotifications` at the top level (not scoped to the Menu being open), so the subscription stays in sync with settings regardless of UI state, and passes `notificationSettings`/`onUpdateNotificationSettings`/`pushStatus` down to `Menu` as props.

Server-side (`server/server.js`): subscriptions persist to `subscriptions.json` (same `DATA_DIR`-aware pattern as `messages.json`), keyed by `subscription.endpoint`. `POST /push/subscribe`/`/push/unsubscribe` are token-checked like `/send`. `pushToSubscribers(type, payload)` filters subscribers by `types[type] !== false`, sends via the `web-push` package, and prunes any subscription the push service reports as gone (404/410) â€” fire-and-forget from `/send` and `/push/notify-update`'s request handlers (not awaited before responding), with each individual send wrapped in its own try/catch so one dead subscription can't take down the `Promise.all` for the rest. `PUSH_ENABLED` (`Boolean(VAPID_PUBLIC_KEY && VAPID_PRIVATE_KEY)`) gates actual sending but not subscription storage, so client and server rollout order doesn't matter.

`src/sw.ts` handles the actual `push`/`notificationclick` events â€” see PWA setup below for why this required switching off `generateSW`.

### PWA setup

`vite-plugin-pwa` is configured in `vite.config.ts` with an inline manifest (name, theme colors, icons) pointing at `public/favicon.svg`. `registerType: 'autoUpdate'` is used so updates apply without prompting. If you change the app icon, only `public/favicon.svg` needs updating â€” it's reused for both the favicon and the manifest icons.

Uses the `injectManifest` strategy (`strategies: 'injectManifest'`, `srcDir: 'src'`, `filename: 'sw.ts'`), not the default `generateSW` â€” push notifications need a custom `push`/`notificationclick` handler in the service worker itself, which `generateSW`'s auto-generated worker has no hook for. `src/sw.ts` calls `precacheAndRoute(self.__WB_MANIFEST)` (from `workbox-precaching`) to keep the same offline-asset precaching `generateSW` used to provide automatically, plus `clientsClaim()`/`self.skipWaiting()` for the same "activate immediately" behavior `registerType: 'autoUpdate'` implies. **Gotcha:** `self` needs `declare let self: ServiceWorkerGlobalScope & { __WB_MANIFEST: ... }` at the top of `sw.ts` alongside `/// <reference lib="webworker" />` â€” the file lives under `src/` and is type-checked by the same `tsconfig.app.json` as the rest of the DOM-based app code, so without the explicit re-declaration, `self` resolves to the ambient DOM `Window` type instead of `ServiceWorkerGlobalScope`, and `__WB_MANIFEST` (a workbox/vite-plugin-pwa build-time convention, not a real DOM/webworker-lib type) doesn't exist on either. `src/sw.ts` is excluded from Vitest coverage (`vite.config.ts`'s `test.coverage.exclude`), same category as `main.tsx` â€” it's bootstrap/platform glue that runs in a completely different execution context, not something jsdom can meaningfully exercise.

### Releases

`useNativeUpdate.ts` is the consumer of a release: it polls `GET /repos/XxFireBoomxX/catmagochi/releases/latest` on GitHub, compares `tag_name` (minus its `v` prefix) against `__APP_VERSION__`, and if newer, downloads that release's `bundle.zip` asset via `CapacitorUpdater.download`/`.set` â€” this is the Capacitor OTA update path for the installed native shell, so a release doesn't mean rebuilding/reinstalling the APK, just publishing a new `bundle.zip` for the already-installed app to pull. The asset name (`bundle.zip`) and the `v`-prefixed tag format are load-bearing â€” both are hardcoded in `useNativeUpdate.ts`.

To cut one:

1. Bump `version` in `package.json` (and run `npm install --package-lock-only` to sync `package-lock.json`) on its own `chore/bump-version-X.Y.Z` branch, PR, merge â€” same pattern as every other change here, version bumps included. This repo bumps the patch digit per release regardless of how big the change set is (e.g. 1.0.6 covered push notifications *and* growth progress *and* the stats window) rather than following strict semver.
2. **Build with `.env` out of the way.** Vite inlines `import.meta.env.VITE_*` as literal strings at build time, and `.env` may hold a real `VITE_RELAY_TOKEN` for local relay testing â€” building normally would bake that secret straight into `bundle.zip`, a *public* GitHub release asset. Move `.env` aside (`mv .env /tmp/... `), run `npm run build`, then move it back. This is the same class of mistake as the sender.html/VAPID-key incidents elsewhere in this project's history: building for a public artifact with real local secrets present.
3. Zip the *contents* of `dist/` (not the `dist/` folder itself) into `bundle.zip` at the repo root â€” `zip`/`unzip` aren't guaranteed to be installed; Python's `zipfile` module works as a fallback. Grep the built output (and ideally the zip contents directly) for anything secret-shaped before uploading, as a last-resort check on top of step 2.
4. `gh release create vX.Y.Z bundle.zip --title vX.Y.Z --notes "..."` â€” tag and release in one step. `bundle.zip` is gitignored (it's a build artifact, published as a release asset, never committed).
5. If the `update` push notification type matters for this release (see "Push notifications" above), trigger it manually per `server/README.md`'s `/push/notify-update` â€” only meaningful once a real relay is actually deployed and reachable; skip it if there isn't one yet.

### Testing

Frontend tests are Vitest + Testing Library, colocated as `*.test.ts`/`*.test.tsx` next to the source file they cover. Vitest config lives inside `vite.config.ts`'s `test` block rather than a separate `vitest.config.ts`, specifically so it shares the rest of that file (e.g. the `__APP_VERSION__` define `useNativeUpdate` needs). `src/test/setup.ts` wires up `@testing-library/jest-dom` matchers and, in a global `afterEach`, calls RTL `cleanup()`, clears `localStorage`, and resets fake timers/global stubs/mocks â€” so tests don't need per-file teardown boilerplate for those.

- Anything timer-driven (`AsciiCat`'s blink/idle loops, `useFlavorText`) is tested with `vi.useFakeTimers()` plus `Math.random` stubbed to a fixed value (usually `0`), then asserted against exact millisecond advances â€” see any of those `.test.tsx` files for the pattern.
- `App.test.tsx` defines a local `renderApp()` helper (`render(<App />)` plus advancing fake timers past `StartScreen`'s `START_TOTAL_MS`) and uses it in place of a bare `render(<App />)` everywhere except its own `start screen` describe block, which needs the raw, pre-advance render to assert on the splash itself.
- `App.test.tsx` mocks `useMessages`, `useCareEvents`, and `useFlavorText` outright rather than exercising their real internals (those are covered by their own dedicated test files) â€” this also sidesteps a real gotcha: this repo's gitignored `.env` may have real `VITE_RELAY_URL`/`VITE_RELAY_TOKEN` values set for local relay testing, and Vite/Vitest auto-loads `.env`, so any test asserting on "relay not configured" behavior must stub those env vars explicitly rather than assume they're unset (see the empty-string stubs in `useMessages.test.ts`). The `useCareEvents` mock captures the `onEvent` callback `App.tsx` passes in, so tests can simulate a remote device's action arriving by just calling it directly, without a real WebSocket.
- `server/` has its own, separate suite (`server/server.test.js`, run via `node --test`, not Vitest) since it's a standalone Node package with no framework dependency â€” it spawns the real `server.js` as a child process against a temp `DATA_DIR` and drives it over actual HTTP/WebSocket rather than mocking `http`/`ws`. Vitest's `test.exclude` skips `server/**` so the two runners don't collide over the same `*.test.js` files.
- Coverage thresholds (lines/statements/branches/functions, all 90%) are enforced in `vite.config.ts`'s `test.coverage.thresholds` â€” `npm run test:coverage` exits non-zero if any drop below that.
